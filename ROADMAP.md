## Phase 1: Foundation Fixes (Weeks 1–2)

These must be done before anything else. They fix what's broken and make existing features actually usable.

### 1.1 — Fix Team Access RLS Gap

> **Why:** Team members you invite currently see nothing. Every business query filters by `user_id = auth.uid()` which excludes invited accountants/staff.
> **Scope:** Create `getAccessibleBusinessId()` helper. Update all 12+ business files. Update RLS policies on all business tables.
> **Status:** ✅ Completed (12 Apr 2026)

### 1.2 — Notifications System

> **Why:** Without this, every feature is invisible. A user gets a comment, a payment, an overdue invoice — they'll never know unless they check manually.
> **Scope:** `notifications` table. `NotificationBell` component in AppNav. Trigger notifications from key events (invoice paid, team invite, comment, follow, overdue). Badge count. Mark as read.
> **Status:** ✅ Completed (12 Apr 2026)

### 1.3 — Loading & Error Boundaries

> **Why:** Users currently see white screens when pages load or queries fail.
> **Scope:** Add `loading.tsx` (skeleton screens) and `error.tsx` (friendly retry UI) to every route group. Create shared skeleton components.
> **Status:** ✅ Completed (12 Apr 2026)

### 1.4 — Wire LinkedIn OAuth

> **Why:** Already configured in developer portal. Quick win for professional users.
> **Scope:** Enable provider in Supabase, add button to login/signup pages, test callback.
> **Status:** ✅ Completed (12 Apr 2026)

### 1.5 — Wire Facebook OAuth

> **Why:** Same as LinkedIn. Major auth provider in Nigeria.
> **Scope:** Enable provider in Supabase, add button to login/signup pages, test callback.
> **Status:** ⏸️ Blocked — Meta Business Verification required; proceeding without (12 Apr 2026)

## Phase 2: User Experience (Weeks 3–4)

Make the platform feel professional and work for Nigerian SMEs on mobile.

### 2.1 — Onboarding Wizard

> **Why:** New users land on an empty dashboard with no guidance. Nigerian SMEs encountering business software for the first time need hand-holding.
> **Scope:** 3-step wizard after first login: (1) Complete profile, (2) Choose modules, (3) Set up first business OR explore directory. Progress indicators. Contextual tooltips on first visit to each module.
> **Status:** ✅ Completed (12 Apr 2026)

### 2.2 — Invoice PDF Generation & Sharing

> **Why:** SMEs need to send professional PDF invoices via WhatsApp and email. Current print layout is good but not a shareable PDF.
> **Scope:** Server-side PDF generation. Branded template with logo, bank details, QR code. Download button. Email as attachment via Resend. WhatsApp share link.
> **Status:** ✅ Completed (12 Apr 2026) — PDF download (html2canvas + jsPDF), WhatsApp share, email share. Expenses search bar + column sorting added.

### 2.3 — PWA / Offline Capability

> **Why:** Many target users have unreliable internet. A farmer at a rural market needs offline access.
> **Scope:** Service worker, manifest.json, install prompt. Cache business dashboard, products, customers for offline viewing. Queue creates/updates for sync.
> **Status:** ✅ Completed (12 Apr 2026) — @ducanh2912/next-pwa, web manifest, PWA icons (192/512), service worker with NetworkFirst for API + CacheFirst for static assets. Install-to-home-screen enabled.

### 2.4 — WhatsApp Notifications

> **Why:** Email open rates in Nigeria ~15-20%. WhatsApp >90%. Your users live on WhatsApp.
> **Scope:** WhatsApp Business API integration (Twilio / Africa's Talking). Opt-in from profile. Send invoice PDFs, payment confirmations, overdue alerts via WhatsApp.
> **Status:** ⏸️ Blocked — Requires Meta Business Verification (same blocker as Facebook OAuth). WhatsApp share links already available on invoices. Revisit after Meta approval or via Termii.

### 2.5 — API Rate Limiting

> **Why:** Public API routes (invite, contact, auth) have no rate limiting. Abuse risk before launch.
> **Scope:** IP-based rate limiter on all public and sensitive API routes.
> **Status:** ✅ Completed (12 Apr 2026) — In-memory rate limiter on 10 routes: reset-password (3/min), contact (5/min), waitlist (5/min), apply (5/min), follow (20/min), invite (10/min), subscribe (5/min), payment (5/min), like (30/min), notifications (20/min).

## Phase 3: Growth Features (Weeks 5–8)

Features that drive daily engagement and create network effects.

### 3.1 — Mentorship Module

> **Why:** High-value networking feature. Creates sticky relationships. Already in middleware matcher.
> **Scope:** Mentor profiles with expertise/availability. Mentee matching. Request workflow (pending/accepted/declined/withdrawn/completed). Session scheduling with date, format, duration, meeting link. Post-session star ratings + reviews (both parties). Notification deduplication. State-machine enforcement via DB trigger. Party-scoped RLS on sessions + reviews.
> **Status:** ✅ Completed (14 Apr 2026) — Mentor profiles, browse/search with expertise + availability filters, request workflow, session scheduling UI (mentor schedules → both see details), mark completed / cancel session, star ratings + review modal, notification link fix, transition enforcement trigger, deduplicated notifications (DB trigger only), tightened RLS on mentorship_sessions + mentorship_requests.

### 3.2 — Smart Price Intelligence

> **Why:** Transforms the price tracker from a data dump into a daily-use tool.
> **Scope:** Price trend charts per commodity/state. Cross-state comparisons. Price alerts ("notify me when maize drops below ₦X in Oyo"). Weekly market digest.
> **Status:** ✅ Completed (12 Apr 2026) — Recharts-powered price trend charts (avg/min/max), cross-state comparison bar charts with auto-insights, price alerts CRUD with live notification triggers, poster attribution on cards. Tab-based UI (Reports / Intelligence).

### 3.3 — Grant Application Tracker

> **Why:** Unique value prop. Turns AgroYield into an indispensable tool for anyone seeking agri funding.
> **Scope:** `/grants` module. Track applications (draft → submitted → shortlisted → rejected → awarded). Document checklist with defaults. Auto-populate from profile. Deadline reminders via notifications. Admin grant posting. Category/status filtering. How-to-use guide.
> **Status:** ✅ Completed (12 Apr 2026) — Full grant listing, application tracker with status pipeline, document checklist (6 defaults + custom), auto-populated profile info, deadline reminders API, admin post form, category + status filters, My Applications dashboard with stats.

### 3.3b — Community Feed

> **Why:** Daily engagement driver. Creates social stickiness — users return to check discussions, vote on polls, and celebrate milestones. Positioned as the #1 card on the dashboard.
> **Scope:** `/community` module. 5 post types (discussion, question, poll, news, milestone). Poll voting with results. Like/comment using existing infrastructure. Post detail page. Filter by type. Pin support.

## Phase 4: Polish & Launch (Weeks 9–12)

Harden the platform, add differentiators, test with real users.

### 4.1 — Marketplace Escrow (Basic)

> **Why:** Capture transactions on-platform. Revenue via commission. Build trust.
> **Scope:** "Request to Buy" flow. Paystack escrow. Buyer pays → seller ships → buyer confirms → funds released. Basic dispute flow.
> **Status:** ✅ Completed (16 Apr 2026)
>
> **Delivered:**
> - 3 new DB tables: `seller_bank_accounts`, `marketplace_orders`, `marketplace_disputes` with full RLS
> - Seller bank account setup with Paystack account verification + transfer recipient creation
> - "Buy Now" button on listing detail page with Paystack payment initialization
> - Paystack webhook extended to handle marketplace payments (separate from subscriptions)
> - Full order lifecycle: pending_payment → paid → shipped → completed/disputed/refunded/cancelled
> - Buyer confirms delivery → auto-releases funds via Paystack Transfer API
> - 7-day auto-release cron for shipped orders not confirmed or disputed
> - Dispute system: buyer/seller can raise, admin resolves (favour seller = release, favour buyer = refund)
> - Admin orders/disputes tab with release, refund, and dispute resolution controls
> - 3% platform commission on all transactions
> - Notifications + Slack alerts at every lifecycle step
> - User-facing pages: orders list (buyer/seller toggle), order detail, bank account setup
> - "My Orders" link on marketplace page

### 4.2 — Business Benchmarking (Basic)

> **Why:** "Your profit margin is 18%. Average for poultry businesses in Lagos is 23%." Creates a moat — more businesses = better benchmarks.
> **Scope:** Aggregate anonymous metrics by sector/region/size. Peer comparison on dashboard. Actionable recommendations.
> **Status:** ✅ Completed (17 Apr 2026)
>
> **Delivered:**
> - 3 new columns on `businesses`: `sector`, `state`, `business_size` with CHECK constraints (10 agri sectors, 36 states + FCT, 4 size brackets) + partial indexes
> - "Sector & Classification" section on business setup page with dropdown selectors
> - `/api/business/benchmarks` API: computes per-business KPIs (profit margin, expense ratio, collection rate, revenue) and peer medians. Peer group = same sector, narrowed to same state when ≥3 peers. Uses medians over averages for outlier resilience. Skips zero-activity businesses.
> - `BenchmarkCard` client component on Business Suite dashboard with 3 states: profile completion nudge, "not enough peers" message, or full comparison with directional arrows + actionable insights
> - Graceful degradation: works with zero peers (shows nudge), 1 peer (shows waiting message), 2+ peers (shows full comparison)

### 4.5 — Security Hardening

> **Why:** Before launch, not after.

### 4.9 — Invoice Delivery Charges

> **Why:** Nigerian SMEs routinely bill freight/logistics alongside goods. Without a dedicated line, users were stuffing delivery into line items (distorts product pricing analytics) or into notes (not taxable, not summable).
> **Scope:** `delivery_charge` column on `invoices` (NOT NULL DEFAULT 0, CHECK ≥ 0). Delivery input in new invoice Summary section. VAT calculated on `subtotal + delivery` (matches FIRS treatment of freight as taxable). Delivery row rendered on both invoice detail view and print/PDF.
> **Status:** ✅ Completed (17 Apr 2026)
>
> **Delivered:**
> - Migration `20260417_invoice_delivery_charge.sql` with idempotent `ADD COLUMN IF NOT EXISTS` + non-negative CHECK
> - New invoice page: delivery state + input in Summary, VAT now computed on `preVat = subtotal + delivery`
> - Invoice detail + print pages render Delivery line when > 0; VAT label now reflects actual stored rate
> - Backward compatible — existing invoices default to 0 delivery

### 17 April 2026 (Session 2 — Cron Hardening)

**26. Crons Are Now First-Class Operational Surface**
We closed the gap on what was arguably the most fragile part of the stack: six scheduled jobs running with no uniform logging, no idempotency protection, no kill switches, and three of them not even registered in Vercel's scheduler. After this session: all 6 crons are wrapped in the F2 harness (`runCron()` + `cron_runs` audit rows), all 6 are registered via the root `vercel.json`, and every one is toggle-able from `/admin` via the `settings` table pattern. The harness response shape is uniform, the idempotency keys prevent double-execution within a day/week, and every run — success, skip, or failure — writes a persisted row. **Why this matters strategically:** at launch we'll be running automated emails to real users' inboxes. A bug that double-sends a birthday email to 500 users is a trust event. Idempotency is no longer a "someday" problem.

**27. Admin Audit Log for Settings Saves**
The kill-switch round-trip test today worked end-to-end (toggle OFF in `/admin` → Save → DB persists `'false'` → cron returns skipped envelope). But during verification we had one transient mismatch where the DB showed `'true'` after a save attempt — resolved before becoming systemic. The episode points to a gap worth closing before launch: there's currently no audit record of *who changed what setting when*. At production scale, an admin could disable the digest cron or change pricing and we'd have no forensic trail. **Action:** Post-launch (or during Week 8 integration pass), add an admin audit log entry for every `settings` save showing `key, old_value, new_value, actor_id, changed_at`. Reuse the existing `audit_log` table. Low effort, high operational insurance.

**28. `cron_runs` is a Free Operational Dashboard**
The audit table we just started populating is rich enough to power an admin "Cron Health" card with zero new infrastructure. One SQL query against `cron_runs` returns: last run time per job, success rate last 30 days, median duration, skipped count, failure messages. Post-launch, exposing this in the admin dashboard turns "is the digest broken?" from a guess into a glance. **Action:** Build a tiny `CronHealthTab.tsx` under admin → Email section once we're within 2 weeks of launch. Uses `supabaseAdmin.from('cron_runs').select(...).order('started_at', {ascending: false}).limit(50)`.

**29. Idempotency Keys Expose a Re-Run Pattern Question**
The dailyKey/weeklyKey scheme we used means once a cron runs today, it physically cannot run again until tomorrow — even if an admin *wants* to re-run it (e.g. to re-send a digest after a bug fix, or re-try after a known outage). Today we had to `delete from cron_runs` manually to re-test. For production, we should add an admin "Re-run this job" button that (a) deletes today's row for that job, (b) POSTs to the cron route with a force flag, (c) audit-logs the manual re-run. **Action:** Build after launch, during the Week 8 integration pass. Low priority but saves Okoli from SQL when things break.

### 17 April 2026 (Session 3 — F1 Public Business URLs)

**30. Migrations Don't Ship — Only Code Does**
The `/b/ag-rentworks` 404 incident exposed a gap we've been living with since day one: Vercel deploys don't run Supabase migrations. Every `supabase/migrations/*.sql` file we commit is source-of-truth documentation, but nothing executes it against prod — the DB changes only when Okoli manually pastes the SQL into the Supabase SQL editor. Today that gap became a production outage: the F1 showcase migration had been merged to the repo, code shipped expecting the 8 new columns, but prod DB didn't have them. `resolveSlug()` destructures `{ data }` without checking `{ error }`, so the Supabase error was swallowed and the route returned `notFound()` silently. **Action (immediate):** After every merge that adds a migration, paste it into SQL editor before declaring the feature shipped. Treat code-deploy and schema-deploy as two steps. **Action (post-launch):** Add `supabase db push` as a GitHub Actions step gated on `supabase/migrations/**` path filter, or a Vercel pre-build hook. Kills this class of bug for good.

**31. Silent Error Swallowing Is a Reliability Smell**
`resolveSlug()` is not alone — the `const { data } = await supabase.from(...)` pattern (without pulling `error`) is probably littered across the codebase. Each instance is a silent 404, empty list, or "feature doesn't work" with no trace in logs. Sentry doesn't see it because it's not thrown. The only way to find out is via user report. **Action:** After launch, add an ESLint rule or grep audit that flags any `const { data }` destructure missing `error`, at minimum on server components and API routes. Second-best is a centralised `supabaseQuery()` wrapper that throws on `.error` — but that's a bigger refactor and has cost/benefit implications for `maybeSingle()` flows.

**34. Production 404s Are Diagnostically Expensive**
Today's debug took ~45 minutes across multiple hypotheses (env vars, service role key, project ref, build cache) before arriving at the actual root cause (missing migration). ### 18 April 2026 (Session 3 — Verified Badge + Business Reviews)

**42. Self-Review Blocking Needs Both RLS and API — Defence in Depth Pays**
Tonight's `business_reviews` migration enforces the "no reviewing your own business" rule at two layers: the RLS INSERT WITH CHECK excludes businesses owned by `auth.uid()`, and `/api/business-reviews` POST does an explicit lookup against `businesses.user_id` before calling insert. ### 18 April 2026 (Session 4 — Recurring Invoices)

**44. JSONB Line-Items Beat A `recurring_invoice_items` Sub-Table**
The initial sketch for `recurring_invoices` had a second table, `recurring_invoice_items`, mirroring `invoice_items` exactly — one row per line. The cron would then have needed a per-template join on every tick to materialise the real invoice. Rejected in favour of a single `line_items jsonb NOT NULL` column on `recurring_invoices` itself. Reasoning: (a) line-items on a template are an atomic unit — you almost never want to query "all templates with a line containing X" the way you'd query real invoices; (b) the cron runs at 06:00 UTC daily against a partial index `WHERE status='active' AND next_run_on <= today`, and every avoided join buys headroom as templates grow; (c) validating a JSONB array at the API boundary (`sanitiseLineItems` enforces shape + 50-row cap + numeric bounds) is cheaper than enforcing the same via CHECK constraints across two tables. The materialised row — the actual `invoices` + `invoice_items` pair — is still normalised; only the *template* is denormalised. **Action:** Apply the same rule to the upcoming Cooperatives feature: the co-op `settings` blob (fee-split rules, pooled-listing preferences) should be JSONB on `cooperatives`, not a `cooperative_settings` child table. Normalise the transactional records, denormalise the templates.

**45. Counter Increment Before Invoice Insert — Avoid Gap-Or-Duplicate Forever**
The cron generator has to mint an `invoice_number` for every materialised invoice. Two failure modes were on the table: (a) increment the business counter first, use that value for the new invoice — if the invoice insert fails, the counter is now one step ahead and the next run gets a gap; (b) insert the invoice with a placeholder, then increment the counter, then UPDATE — if anything fails mid-sequence you either duplicate or gap. Chose (a) deliberately. The invoice insert itself is idempotent-ish (the daily cron's `idempotencyKey: dailyKey()` ensures the cron row runs at most once per day, and a mid-run crash on a specific template just means the next day's run — or a manual retry — picks up from the bumped `next_run_on`). A gap in invoice numbers is a visible but cosmetic artefact; a duplicate is a real customer-facing bug (two invoices with identical numbers can't be paid separately in accounting software). **Action:** Any future auto-numbered artefact (purchase orders, receipts, credit notes) must follow the same rule: increment-then-insert, accept rare gaps, never allow duplicates. If gap-sensitivity matters later (e.g. tax auditors query continuous sequences), introduce a `gap_audit` view that lists missing numbers per business per month rather than trying to make the increment-insert atomic.

**46. Paused-For-A-While Recurring Templates Need Resume Bump-Forward**
The Resume button on `/business/invoices/recurring` flips `status` from `paused` to `active`. The subtle bug it fixes: a template paused for two months has a `next_run_on` that's eight weeks in the past. Without intervention, the next cron tick would generate eight backlogged invoices — exactly the anti-pattern the Pause button exists to prevent. `/api/recurring-invoices` PATCH on action=resume now checks `next_run_on < todayUtc()` and bumps it to today before saving. Customers see "resumed this month", not "surprise invoice avalanche". **Action:** Any scheduled-generator feature gets the same treatment — recurring AI digest emails, scheduled report generation, scheduled price-alert sweeps. The invariant is "resume means next_run_on ≥ today, always".

**47. Feature-Flag-Off Must Still Bump Schedule — Don't Burn Daily Cycles**
When the `recurring_invoices` feature flag is off for a specific user (via the 4-week rollout bucket, allowlist, or admin kill-switch), the cron needs to skip them — but *without* leaving `next_run_on` where it is. If we just skip, the same template gets re-queried every single day, forever, burning query budget against the same row until someone either enables the flag or ends the template. The fix: `bumpScheduleNoInvoice()` advances `next_run_on` by one cadence step even when no invoice gets generated, writes a `last_error = "Feature disabled for this business"` note so the UI surfaces why, and the cron moves on. Same pattern applies to tier downgrades (dropped to Free → note says "Upgrade to resume recurring invoices", schedule still bumps). **Action:** Any cron-driven feature with per-row gating needs the two-branch pattern: gated-and-generate vs gated-and-skip-but-bump. Silent "skip and leave row hot" is a slow-motion resource leak.

**50. Quota Increment Belongs AFTER The Expensive Work, Not Before**
Building Unicorn #5 Expense OCR. Four gates layered on POST `/api/expense-ocr`: rate-limit → auth → flag → ownership → **tier quota** → file-validation → storage upload → Vision call. The interesting design choice is where the counter increment lives. Two options: (a) increment before calling Vision — prevents an abusive user from making N concurrent calls that all pass the quota check because none of them have incremented yet; (b) increment after a successful extraction — a failed Vision call (bad image, API 500, model refusing to parse) doesn't burn a quota slot. Chose (b) deliberately. The rate-limiter (5/min/user) caps the concurrent-abuse window to a bounded loss even if two requests race. The UX win is significant: a user who gets a blurry-photo failure on their 20th-of-the-month scan isn't locked out for April over a model error. Concretely: `checkQuota` reads the current count before the Vision call; `incrementUsage` only runs after `expense_ocr` insert succeeds with a valid extraction. Pre-extraction rejections (validation errors, flag off, tier gate, kill switch) also don't count, by construction. **Action:** Any quota-gated feature where the underlying work can fail for non-user reasons (Vision, AI Assistant, Credit Score refresh) must use the read-check-work-increment pattern, not read-check-increment-work. Also: quota rows use `period_yyyymm` as a unique-constraint key, which means natural monthly rollover — a new month just upserts a new row starting at count=1. No reset cron is strictly needed; the usage-reset cron is purely for observability + retention pruning (prune rows older than 12 months).

**49. Feature-Flag-Default-False + Fire-And-Forget UI = Phantom Success**
The first session-4 hotfix (FK migration + page-level error surfacing) didn't actually fix Okoli's reported bug. After paste, the recurring page still showed empty — because the rows weren't in the table at all. SQL diagnostic confirmed: `select count(*) from recurring_invoices` returned `0`. The real cause: `recurring_invoices` is one of the 8 unicorn flags seeded as `is_enabled = false` in `20260417_feature_flags.sql` (correct, deliberate — staged-rollout default). When the user ticked "Make this recurring" and submitted, `/api/recurring-invoices` POST returned 403 with `{ error: "Recurring invoices aren't available for your account yet" }`. The new-invoice page's POST was wrapped in `try { await fetch(...) } catch { /* non-blocking */ }` — but `await fetch()` only throws on network errors; HTTP 4xx/5xx resolves normally with `res.ok = false`. The bare catch caught nothing. User got a regular invoice + zero indication their recurring schedule didn't save. Two failure modes compounded: (a) the gate's polite refusal is invisible, (b) the form's optimism papers over the refusal. **Two fixes shipped:** (1) `update feature_flags set is_enabled = true where key = 'recurring_invoices'` — the immediate unblock for Okoli's account; (2) the new-invoice POST now checks `res.ok`, reads the JSON body for the server's error message, stashes a human-readable warning in `sessionStorage.recurring_create_warning`, and a new `RecurringCreateWarning` client component on the invoice detail page renders it as a dismissible amber banner with a link to `/business/invoices/recurring`. Network-level throws follow the same surface path. **Two go-forward rules:** (1) Any "non-blocking" fetch that fires after a successful primary write must still surface non-2xx responses to the user — the primary write succeeding doesn't justify silent failure on the secondary. The pattern is "best-effort with visible failure", not "fire and forget". (2) Default-false feature flags need an admin-UI toggle that doesn't require SQL editor access. **Discovery during the hotfix:** this UI already ships — `/app/admin/tabs/FeatureFlagsTab.tsx` (super-admin-only, wired via `admin-client.tsx`, PATCH `/api/admin/feature-flags` with `logAdminAction` audit entries). It supports per-flag quick toggle, rollout percentage (0-100 slider, step 5), user allowlist, business allowlist, Save/Discard with toast, 60s propagation note. The `update feature_flags set is_enabled = true ...` SQL Okoli ran could have been one click in `/admin` → Feature Flags → toggle `recurring_invoices`. **Real go-forward rule:** when shipping a new default-false flag (next is probably #5 Expense OCR), the rollout checklist must include "confirm flag is listed in `/admin` Feature Flags tab" — the tab reads all rows from `feature_flags`, so a new insert shows up automatically, but we should click through the flow once per flag to verify. And the hotfix note that suggested building the UI was wrong; the failure mode was discoverability, not missing tooling — Okoli wasn't aware it existed. Document it in the admin runbook.

**48. PostgREST Nested Selects Need FKs — And A Missing FK Silently Hides Rows**
Two recurring invoices created. `/business/invoices/recurring` rendered the empty state anyway. Root cause: the `20260418_recurring_invoices.sql` migration declared `customer_id uuid NOT NULL` without `REFERENCES customers(id)`. The page query used `.select('... customers(name)')` — PostgREST's nested-select syntax that requires a foreign-key relationship to resolve the embed. Without the FK, Supabase returned "Could not find a relationship between 'recurring_invoices' and 'customers' in the schema cache", the page discarded the error (only `data` was destructured, not `error`), and `data` fell to null → rows.length === 0 → empty state. The `invoices` table has the same gap in the baseline migration but appears to have had the FK added manually via the Supabase dashboard at some point, which is why the existing invoices page works. **Two fixes shipped:** (a) `20260418_recurring_invoices_fks.sql` idempotently adds FKs for `customer_id → customers(id)` (ON DELETE RESTRICT so you can't orphan live templates), `user_id → auth.users(id)` (ON DELETE CASCADE), plus re-asserts `business_id → businesses(id)`; ends with `NOTIFY pgrst, 'reload schema'` to refresh the PostgREST cache without a restart; (b) the page is rewritten to a two-query pattern (fetch recurring_invoices, then fetch customers(id, name) .in() by the collected IDs) instead of relying on the embed. This is defence-in-depth: even if a future migration breaks the FK, the page still renders real rows. **Two rules going forward:** (1) every `uuid` column that names a parent table — `customer_id`, `business_id`, `user_id`, `product_id` — must have an explicit `REFERENCES parent(id)` clause in the same migration that creates the column; never inherit from "the baseline probably has it". (2) Any server-rendered page query must destructure `error` and render it visibly; silent null-fallthrough converts schema bugs into "feature appears broken" bug reports instead of actionable stack traces.

### 18 April 2026 (Session 5 — Pre-Beta Hardening)

**51. `useState(initFn)` + `Date.now()` In A Client Component = Hydration Bomb That Fires On Every Visit**
The lone Sentry issue on the 24h dashboard turned out to be "Hydration failed — the server rendered HTML didn't match the client" with 4,211 events on `https://agroyield.africa/` — i.e. the landing page was breaking hydration for essentially every visitor, not some rare edge. Root cause in `app/home-client.tsx`: the `Countdown` component's initial state was `useState(calc)` where `calc` read `Date.now()` inline. React ran `calc()` once during SSR (server clock), then ran it again during the client's initial render for hydration (browser clock). Because the seconds field always differs between the two moments in wall time, the server HTML and the client-rendered markup diverged on every request. To make things worse, the launch-date literal `new Date('2026-07-05T00:00:00')` — no `Z` suffix — parsed as UTC on the server and LOCAL time in the browser, so even the days/hours/minutes would drift by the user's timezone offset, compounding the per-second mismatch into a per-timezone mismatch. **Fix shipped:** (a) moved the launch instant to module scope with an explicit `Z` suffix so both runtimes resolve the same absolute millisecond; (b) extracted `calcTimeLeft()` to a pure module-scope function; (c) initialised `useState<TimeLeft | null>(null)` — null on SSR and null on the client's first paint is identical markup by construction, so hydration cannot disagree; (d) started the interval + first real calculation inside `useEffect`, which by spec only runs after hydration completes; (e) rendered a `—` placeholder for each unit while `time === null`. `tsc --noEmit` clean post-fix. **Three go-forward rules:** (1) Any `useState(initFn)` where `initFn` reads a side-effectful global (Date.now, Math.random, window.\*, localStorage, sessionStorage, document.cookie, navigator, location) is a hydration bomb — the pattern is `useState(<static value>)` + `useEffect(() => setState(compute()), [])`. (2) Every date/time literal in the codebase must end in `Z` (UTC) or an explicit offset like `+01:00`; bare `'2026-07-05T00:00:00'` strings are timezone-ambiguous and will parse differently on server vs browser the moment the hosting region and the user's browser disagree. (3) Hydration errors should be treated as pre-Beta blockers even when the UI "looks fine" — Sentry was showing 4,211 events from ONE bug on ONE page, which would have drowned every real error in noise during the launch window. Before Beta goes live, a grep audit for `useState(<identifier>)` (no parentheses, so referencing a function) and `new Date('[0-9]'` without `Z` is worth running.

**52. "Investigate The Lone Sentry Issue" Was The Highest-ROI Pre-Beta Task This Week**
We nearly shipped to Beta with this bug live. The Sentry dashboard showed exactly one issue in 24h — "only one" sounded reassuring, so the instinct was to de-prioritise it behind the rollout runbook and welcome email. The event count (4,211 in 24h against maybe low-hundreds of unique visitors) was the tell: it wasn't a rare edge case, it was every visitor hitting the same bomb every second the page stayed mounted. **Two go-forward operational rules:** (1) Sentry issue counts need to be read as `events / unique_users`, not raw event totals — a single bug producing thousands of events on a traffic base of dozens is a 100%-reproducibility bug, full stop. **53. Settings-Backed Constants With SAFE_DEFAULTS + 60s Cache + Write-Time Validators Is The Right Pattern For Any Tunable The Founder Will Want To Change Post-Launch**
Going into the Pre-Beta hardening pass, the instinct was "we'll fix these after the first week of real data". A grep across the codebase for hardcoded thresholds, tier quotas, category lists, and copy strings surfaced ~dozen places where a post-launch tune would require a code change + redeploy — meaning a round-trip to Cursor for something the founder absolutely should be able to do from the admin UI at 11pm on a Tuesday. The two highest-risk ones (the Free/Pro/Growth OCR + AI Assistant monthly quotas in `lib/usage-tracking.ts`, and the 10-entry expense categories list that both the OCR Vision prompt AND the Expenses page filter pills read) got the full treatment this session. **The pattern that emerged is worth codifying:** (a) seed sensible defaults into `settings` via an idempotent migration with `ON CONFLICT (key) DO NOTHING` so first deploy is byte-identical to the pre-migration hardcoded values; (b) keep a `SAFE_DEFAULTS` constant in the library that mirrors the seed exactly, so the reader fails open when the row is missing or malformed (billing-adjacent features must not blank out on a typo — scratchpad #12); (c) async getter with 60s in-memory cache (mirrors `lib/feature-flags.ts`) so enforcement is live without hammering the DB on every request; (d) `__clearCache()` helper exported for tests and for forcing propagation after admin edits; (e) write-time validator per-key in `/api/admin/settings` that rejects the ENTIRE batch (400) if any single value is malformed, so the admin sees a clear error instead of a partial save that silently falls back to defaults (scratchpad #18 in full flight); (f) Option-B confirmation modal on the admin UI for changes with mid-month blast-radius (quota tightening can lock out users who already committed spend this month, so explicit acknowledgement of the diff is worth the extra click). **Client-vs-server caveat:** any `'use client'` file cannot import the server-only settings library directly (would pull the supabase admin client into the browser bundle). Solution: extend an existing public API route (we used `/api/content-types`) to also serve the dynamic value, and fetch it in a mount-time `useEffect` with the SAFE_DEFAULTS rendered during the loading window. SSR markup must match the client's first paint by construction (scratchpad #51), so the initial state must be the static fallback, never the dynamic value. **Action for the rest of the grep hits** (pricing labels, payment grace period copy, social links, sender addresses, countdown launch date, cron schedule windows, etc.): work through them post-Beta in priority order — the rule of thumb is "would Okoli plausibly want to change this in the next 90 days without a deploy?" ### 19 April 2026 (Session 6 — Production Deploy + Live Validation)

**54. Bucket A/B/C/D Framing Is The Right Mental Model For Admin-Control Audits — And The D Bucket Matters As Much As A/B/C**
The session-5 grep pass that surfaced ~dozen hardcoded constants wanted to be structured, not a laundry list. The framing that landed — and should be reused for every future admin-control audit — is a four-bucket sort: **A = already admin-controlled pre-audit** (find these so you don't re-promote things that already work; also catches surfaces where the DB write path exists but no UI was ever built); **B = primary batch worth promoting now** (highest-ROI, usually tier quotas or copy strings, ship in one commit with shared pattern); **C = bonus promotions that fit the same pattern with minimal extra work** (ship in the same commit if tractable, defer only if they change the shape of the migration); **D = rejected on security/stability grounds, never to move**. The common mistake is to treat D as "didn't get to it" — actually it's the bucket that prevents the audit from becoming a liability. In this codebase: `ANTHROPIC_API_KEY`, Supabase service role, Resend key, Sentry DSN, PostHog key, webhook secrets — ALL stay in Vercel env, NOT in `settings`, because env vars are a stronger trust boundary than a row in a PG table that could theoretically be reached by anyone who can execute arbitrary SQL through a future bug. Email `from:` addresses are another D — moving them to `settings` would let any attacker who compromises `settings` redirect transactional mail to an attacker-controlled destination. **Two go-forward rules:** (1) every admin-control audit produces four lists, not one; the D list is written down and justified; (2) "would the founder want to change this in the next 90 days without a deploy?" is necessary but not sufficient — if the answer is "yes" but the data flows from `settings` to an external integration in a way that enables data exfiltration on compromise, it's still D.

**55. Real-User-Real-Device Validation Beats Headless Smoke Scripts For Mobile-First Products**
`scripts/smoke-test-journeys.sh` passing 30/30 against production gave us false confidence that the Expense OCR path was validated end-to-end. It wasn't — all the headless script can prove is that HTTP routes return 2xx. It cannot exercise: camera permission prompts on iOS/Android, the `capture="environment"` attribute actually opening the rear camera vs. the photo library, real image dimensions and file sizes from a phone's actual sensor (~4-8 MB is common, up against our 5 MB cap), real MTN/Airtel 4G upload latency through Nigeria's network layer, OS integration friction like the user backgrounding the app mid-upload, touch-target sizing on small screens. **57. Destructive Actions Inside Nested-Module Layouts Need In-Module Affordances, Not Round-Trips To The Parent Hub**

### 20 April 2026 (Session 8b — Red-Team Probe On Business-Registration Duplicate Prevention)

**61. Slug Decoration Is Not Duplicate Prevention — And Identity-Bearing Columns Need An Explicit Uniqueness Decision At Schema-Design Time**
Okoli deliberately inserted a second business row with near-duplicate branding — "PREEMINENT SOLUTIONS" (slug `preeminent-solutions`) alongside the pre-existing real "Preeminent Solutions Nig Ltd" (slug `preeminent-solutions-nig-ltd`) — to probe whether the registration pipeline would detect the collision. Both rows landed cleanly. The failure mode is subtle: the existing `ux_businesses_slug` unique index plus the `businesses_auto_slug` trigger did exactly what they were designed to do — make the URL slug unique — but that design choice silently resolves what should have been a hard stop. In this specific probe the trigger's decoration path wasn't even exercised, because the two different source names produced naturally-distinct slugs without any collision for the trigger to resolve; but even when decoration DOES fire (e.g. a second submission of the exact name "Preeminent Solutions" would become `preeminent-solutions-2`), the effect is the same — the database accepts a semantic duplicate without a single signal reaching the application layer. No PG 23505, no API 409, no admin notification, no audit-log row. The failure is invisible to every tier of the stack. **Two generalised lessons.** **(a) Auto-resolving a unique-constraint violation via decoration is an anti-pattern when the signal that actually matters is semantic uniqueness, not URL uniqueness.** URL-slug decoration is correct behavior for a blog's post titles (two posts titled "Year In Review" should both be publishable under distinct permalinks). It's wrong behavior for entities that represent real-world things — companies, users, physical locations — where the duplicate carries business meaning. The decoration *hides* the signal that should have fired. **Rule:** when a UNIQUE constraint exists on a derived field (slug, short code, handle) whose only purpose is to disambiguate URLs, pair it with a separate uniqueness-or-detection pass on the semantic identity field (CAC number, tax ID, legal name + jurisdiction) BEFORE the derived-field collision resolver runs. **(b) Every column that carries real-world identity needs an explicit uniqueness decision at schema-design time, even if the decision is "no uniqueness because X."** On `businesses`, the identity-bearing columns are `cac_number`, `vat_tin`, `phone`, `email`, `whatsapp`, and `name`. None of them have unique constraints, indexes, or documented rationale for why they don't. The default state of "no constraint = no intent" is indistinguishable from "no constraint = oversight" — which is exactly the gap the probe exploited. **Rule:** any schema migration that adds an identity-bearing column must include a COMMENT on the column explaining the uniqueness policy — either `comment on column businesses.cac_number is 'UNIQUE across all businesses — CAC numbers are globally unique in Nigeria'` OR `comment on column businesses.email is 'NOT unique — the same personal email can legitimately own multiple businesses by design'`. Explicit absence is a decision; implicit absence is a bug. **Fix plan codified on the PROJECT_STATUS.md Pending Tasks list:** (1) cheapest/highest-signal — partial UNIQUE on `lower(cac_number)` where not null + not empty, one migration, zero UI risk, shippable pre-Beta; (2) soft-dupe pre-insert check with `ilike` name + state + phone intersection, returning a claim-ownership reconciliation UI; (3) admin-verification-as-sieve via `is_public=false` default. Options 2 + 3 are post-Beta because they touch registration UX and benefit from real Beta traffic to calibrate fuzzy-match thresholds. **Meta-lesson on red-team probes as an engineering practice.** The probe succeeded because it was designed specifically to be invisible to the system — near-duplicate but not exact-duplicate branding, submitted through the actual production registration flow rather than a synthetic test harness, and indistinguishable from an honest typo by a real user. Cost to execute: ~5 minutes. Surfaced an architectural gap that no amount of code review or automated testing would have caught, because code review verifies "does this constraint do what it says" while the gap is "does the absence of this constraint matter". **Go-forward operational rule:** for every new user-facing write path (registration, upload, invoice-create, review-post, support-ticket-open), plan and execute one red-team probe BEFORE launch that tries to produce the outcome you explicitly don't want — duplicate entity, privilege escalation, data leak across tenants, rate-limit bypass — and measure whether any layer of the stack detects the attempt. A probe that produces silence is more informative than a passing test suite. Add to the pre-rollout runbook alongside scratchpad #58's `grep -rn 'href="#"' app/` check.

**62. `overflow-hidden` Wrapper Is Non-Negotiable For Avatars — And The 3-Instance Rule Tightens When The Rendered Thing Is Identity-Bearing And Sized Differently Across Contexts**
Okoli reported "the image avatar in the business module tend to allow an overflow of the image" on the business dashboard header. Typecheck clean. **Three generalised lessons.** **(a) `overflow-hidden` on the wrapper is the ONLY bulletproof clip for image-based avatars.** Applying `rounded-*` to the `<img>` element itself relies on the image's intrinsic content respecting the CSS box radius — which it mostly does, but not reliably under zoom, `object-*` fit modes, or non-square source aspect ratios. Writing this as a rule: any clipped image element should ALWAYS be inside an `overflow-hidden` wrapper, and the wrapper's fixed dimensions are the clipping source of truth, not the image's width/height props. **(b) The 3-instance rule (scratchpad #56) tightens when the rendered thing is identity-bearing and sized differently across contexts.** The business-logo anti-pattern existed in exactly 2 places today, below the 3-instance extraction threshold. The 40-line audit that surfaced 7 `logo_url` renderers across the codebase would have to be repeated every time someone adds a new place a business shows up. The easy path was the 2-line wrapper-div fix on `app/business/page.tsx` alone, which would have closed the reported bug with zero collateral work. The correct path was the primitive extraction plus the 4-site migration, which closes the defect CLASS and prevents every future regression — cost ~40 minutes, benefit permanent. **64. Race-Condition-Safe Live Search Belongs In Every Debounced Fetch From Day One — `latestQueryRef` Is The 4-Line Pattern That Prevents Stale-Response Overwrite**
The `GlobalSearchBar` command-palette overlay debounces user input by 250ms then hits `/api/search?q=&limit=5`. Without the right scaffolding, this is a class of bug that's invisible in development (where the network is fast and queries return in their natural order) but produces visible flickers and wrong-result-for-current-query in production with Nigerian 4G latency. The failure mode: user types "lag", debounce fires, request A starts. User types "lago", debounce fires, request B starts. Request B happens to return first (smaller result set, hot in PostgREST cache). UI renders B's results. Request A returns 80ms later carrying the older "lag" results. UI overwrites B's correct render with A's stale data. Without observability of the actual queries, this looks like "the search is just buggy" — there's no error, no retry, just a quietly-wrong result. **The fix that ships in `GlobalSearchBar`:** a `latestQueryRef = useRef<string>('')` records the most recently-dispatched query string. Every fetch's `.then` checks `if (latestQueryRef.current !== thisQuery) return` before calling `setResults`. Stale responses get discarded silently; only the response matching the current query is allowed to render. The pattern is 4 lines and prevents an entire class of flicker bug. **Generalised rule.** Every debounced-fetch surface (search-as-you-type, autocomplete, filter pickers that re-fetch on each keystroke, type-ahead in forms) needs the `latestQueryRef` pattern. This is not a "we'll add it if it becomes a problem" pattern — it's a Day 1 architectural addition because the bug it prevents is invisible-in-dev / visible-in-prod and the cost to add it after the fact is the same as adding it up front. **Pair with two adjacent rules.** (a) **The debounce timeout itself is a per-feature decision** — 250ms feels right for a search-while-typing palette where users are skim-evaluating results, but a filter picker that triggers a full table re-render might want 500-750ms; an autocomplete on a short field (3-letter airport code) might want 100ms. Don't copy a debounce constant across features without re-asking the question. (b) **Every debounced fetch needs a loading-state skeleton, not a blank** — the 250ms gap between keystroke and response is otherwise a perceived freeze on slow connections. `GlobalSearchBar` shows a small "Searching…" affordance during loading, which closes the perceived-freeze gap. **Counter-instance to remember.** The pattern does NOT generalise to non-debounced one-shot fetches (form submits, button clicks) — those are correctly serialised by the user's intent and don't suffer the race. Adding `latestQueryRef` there is dead weight. Use the pattern when and only when keystroke-driven fetches can race; not when user-action-driven fetches can't.

**65. ILIKE Wildcard Injection Escape Is A Cheap Correctness Win That Pays For Itself On The First Underscore-In-Username**
`lib/global-search.ts` exports `escapeIlike(raw: string)` that escapes `\`, `%`, and `_` in user query strings before composing the ILIKE pattern. Without the escape, a user searching for the username "john_doe" would match any 7-character string starting with "john" because `_` is a single-character wildcard in PostgreSQL's LIKE / ILIKE syntax. With ~thousand-row tables it's mostly cosmetic (extra noise in the result set); with the projected Beta + post-Beta growth it's a correctness bug that would surface as user complaints — "I searched for my own username and got someone else's profile first". **The escape adds three lines and zero runtime cost.** The pattern is `'%' + escapeIlike(q) + '%'` instead of `'%' + q + '%'`, plus a regex-replace function defining the three replacements. It's cheap forever; the absence of it is an indignity that grows linearly with table size. **Generalised rule.** Any ILIKE/LIKE pattern composed from user-supplied input needs explicit wildcard escaping at the composition site. The PostgREST `.ilike()` filter does not escape for you (it can't — it has no way of knowing whether `%` is intentional or accidental). **Pair with the broader principle:** every place user input crosses into a pattern-matching DSL (regex, glob, SQL LIKE, full-text search syntax, even `RegExp` constructors in JavaScript) needs explicit escaping at the boundary. Treat each DSL crossing as a security/correctness gate, not as a transparent string concatenation.

The default of "append new widgets to the top because they're most visible" degrades the page for every user who isn't the developer who just added the widget. **Operational rule:** whenever adding a widget to an existing surface with ≥3 widgets, the PR description must include the chosen position AND the rationale ("placed above the form because X is identity-adjacent / placed below because X is secondary analytics"). Without the rationale, the widget will end up at the top by default and the surface accretes friction. **Pair with a design-system rule:** consider introducing a `<PageSections>` primitive that accepts an ordered array of {id, component, priority} entries so the render order is centralised data, not scattered JSX order. Park this as a post-Beta consideration — the win over the Beta horizon is small because we only have one multi-widget surface (`/profile`) plus maybe `/dashboard` down the road, so the abstraction is premature today.

**68. The Timestamptz↔Date-Input Contract Is A One-Line Silent Bug Class That Ships Undetected Until A User Has State In The Column**
`open_to_opportunities_until` is stored as `timestamp with time zone` (per the migration), but the profile form renders it via `<input type="date">` which only accepts `YYYY-MM-DD`. When Supabase returns `2026-06-01T00:00:00+00:00`, the date input silently ignores it and renders blank. The bug is invisible during development if the developer tests with an empty column; it only manifests the first time a user has a saved expiry and revisits the form — they see a blank field and assume it was never saved. **Fix applied:** one-liner `.slice(0, 10)` in the initialData assembly with an inline comment explaining the timestamptz↔date-input contract. **Generalised rule.** Any Postgres `timestamptz` or `timestamp` column that drives an `<input type="date">` needs to be sliced to `YYYY-MM-DD` before being handed to the input. Same for `<input type="time">` (slice after the T, before the timezone); same for `<input type="datetime-local">` (slice before the timezone). HTML form inputs are stricter than browser Date parsers — they silently ignore unrecognised formats rather than coerce. **Pair with two adjacent rules.** (a) **When defining a migration, pause and ask "what UI will read this column?"** If any UI renders it via `<input type="date">` or similar, consider whether the column should be `date` (not `timestamptz`) to sidestep the slicing. The "just store timestamptz for everything" default is tempting but produces this exact class of bug. (b) **Every field with both a DB type and an input type needs a round-trip test** — save a value, reload the page, confirm the field still renders the saved value. This is a 30-second manual check that catches this bug class at development time. The regression test is `fireEvent.change → save → re-render → expect input.value`. Add this to the QA checklist under "any form field with a datatype mismatch".

### 20 April 2026 (Session 12 — JSON-LD Parser-Compat Hotfix, Sentry f2499c29…)

**69. Spec-Valid ≠ Parser-Compatible In The Wild — When You Emit A Wire Format Consumed By A Universe Of Unknown Clients, The Output Shape Must Be The Intersection Of Permissive AND Strict Parser Behaviour, Not Just The Spec**
A single Sentry hit at 07:54 UTC on `/login` (Safari 26.3 on Mac, one occurrence, `mechanism = auto.browser.global_handlers.onerror`) exposed a parser-compat issue in our sitewide JSON-LD: `TypeError: undefined is not an object (evaluating 'r["@context"].toLowerCase')`. Triage isolated the call pattern — some script (browser extension, Safari Reader Mode, SEO toolbar; the exact consumer doesn't matter) does `JSON.parse(script.textContent)["@context"].toLowerCase()` directly on each `<script type="application/ld+json">` block. That pattern works if the root is a single object with a top-level `@context`. It crashes if the root is an array (arrays don't have `@context` → `undefined.toLowerCase()` throws). We had been emitting `JSON.stringify([organizationJsonLd, websiteJsonLd])` as a single script at both `app/layout.tsx` and `app/b/[slug]/page.tsx` (LocalBusiness + BreadcrumbList) — top-level JSON arrays are spec-valid JSON-LD, but naive parsers treat each `<script>` as a single document. **Fix applied.** Split each emission site into two separate `<script type="application/ld+json">` blocks, each holding one schema. Per-site variables renamed for clarity (`organizationJsonLdScript` / `websiteJsonLdScript` in layout; `localBusinessJsonLdScript` / `breadcrumbJsonLdScript` on /b/[slug]). Inline comments at both sites cross-reference the Sentry issue ID (f2499c29…) so the next engineer finds the rationale in-tree. `./node_modules/.bin/tsc --noEmit` EXIT=0. Zero migrations, zero env var changes, zero SEO impact (Google's Rich Results Test accepts both shapes identically). **Generalised rule.** When you emit a wire format consumed by a universe of unknown clients — JSON-LD scripts, webhook payloads, RSS feeds, oEmbed responses, robots.txt, sitemap.xml, OpenGraph meta tags — the output shape must be the *intersection* of permissive AND strict parser behaviour, not just whatever the spec permits. Spec-compliance protects you against "this consumer is correct"; shape-intersection protects you against "this consumer is buggy". The producer is the only party with leverage — consumers are unknown and unfixable. **Pair with an adjacent rule.** Browser-extension / naive-parser crashes show up in Sentry as unhandled errors on URLs you own, even when your code did nothing wrong. When a Sentry issue has (a) single occurrence, (b) single user, (c) `mechanism = auto.browser.global_handlers.onerror`, (d) a shape like `undefined is not an object` on a property your code never reads back, **don't mute — triage for "is there a defensive producer-side change that narrows the failure surface?"** Even if the consumer is third-party, the fix is almost always a one-commit producer-side tightening that improves our compat with the long tail of parsers at zero cost to correct consumers. **Meta-lesson codified.** Scratchpad #66 established defence-in-depth at user-input-to-DB boundaries. #69 extends it to producer-output-to-consumer boundaries — the symmetric case. Both sides of every data boundary are potential correctness gates; the producer side is the one we control, so the fix lives there even when the bug manifests on the consumer side.

### 21–22 April 2026 (Session 15 — Pre-Launch Hardening Drill + WhatsApp Invites + @mentions Scoping)

**70. Pre-Launch Hardening Drill > Pre-Launch Hardening Checklist — A Checklist Confirms Code Exists; A Drill Confirms Alerts Land**
The pre-Beta window surfaced a category distinction between a hardening *checklist* and a hardening *drill*. Checklists confirm an artefact exists — "uptime monitor installed ✓", "Slack webhook wired ✓", "rate limiter configured ✓". They're necessary but insufficient. A *drill* confirms the artefact fires correctly end-to-end — point the rate limiter at itself with 120 req/min and watch for the 429; set a cron job to fail deliberately and watch the Slack channel; swap a health endpoint to return 503 for 30 seconds and watch Better Stack page the on-call. The drill is the load-bearing step, because the *first time a surface fails should not be the first time anyone looks at where the alert goes*. Over the 21–22 Apr window, all five Pre-Launch operational items went through forced-failure drills: rate limiter (probe + Retry-After header verified), Slack `#all-agroyield-alerts` channel (test-fired from Better Stack + cron harness + report-issue endpoint), Better Stack (forced `/api/health` failure → alert landed within 90 s and auto-cleared on recovery), migrations (dry-run on `20260421_price_category_oil_rename.sql` returned 0 rows, confirming no stale state), Termii sender ID (test SMS to Okoli's phone verified `AgroYield` rendering). Every drill produced a byproduct — an updated runbook entry, an inline comment, a scratchpad — so the rehearsal-to-doc arc compounds across future launches. **Generalised rule.** For any production surface whose purpose is to fire an alert when something breaks, the hardening step is not "install it" — it's "install it AND fake the break AND watch the alert land AND time the latency AND confirm the recovery signal". Anything less is a checkbox, not a safeguard. **Pair with three adjacent rules.** (a) **Every operational drill ends in a reusable runbook**, because the rehearsal is the cheapest time to write the "what to do when this fires for real" doc — you have the exact commands in scrollback and the decision tree fresh in memory. Cost of writing the runbook during the drill: ~10 min. Cost of writing it during a real incident: significantly higher, with an adrenaline tax. (b) **Drills should forced-fail every delivery channel at least once before launch** — Slack channel, email, SMS, pager — because each channel has its own failure modes (Slack webhook rate-limiting, email greylisting, SMS sender-ID issues, pager escalation mis-config) and the cross-product of "which channel" × "which failure mode" is where incidents live. (c) **Keep the drill script in the repo** (`scripts/rate-limit-probe.ts`, `scripts/smoke-test-journeys.sh` and so on) so the same rehearsal runs against every future release. Ephemeral shell one-liners used during the drill are the anti-pattern; they're invisible to the next-launch operator. **Meta-lesson.** This extends the scratchpad #55 pattern ("real-user-real-device validation beats headless smoke scripts for mobile-interactive features") to the operational side: real-forced-failure drills beat checklist-level "is it installed". Same underlying principle — verification fidelity scales with how closely the test resembles the real failure — applied across two different surfaces (user-facing UX for #55, operational-alerting for #70).

### 23–27 April 2026 (Session 16 — Beta Launch + Live Hardening Sprint)

**71. Self-Caused N+1 Is The Class You Catch By Auditing Your Own Last-Week's Diff Before Trusting Sentry's All-Clear**
The /community ReportButton N+1 — 40-50 redundant `GET /api/report?postId=X` calls per page load on a 50-post feed — was caused by THIS sprint's own community-Report-button work earlier in the same week. Sentry's `category: 'http_client'` heuristic flagged it as a HIGH severity perf issue (≥10 affected users in the first hour after deploy). The mount-time `useEffect` on `ReportButton.tsx` was correct on a single-post detail page (one extra GET per page-load is fine) and catastrophic on a list page (one extra GET *per visible card*). **Generalised rule.** When a child component fires a fetch on mount, that fetch is correct on detail pages and wrong on list pages by default — the list page must pre-fetch the relevant state and pass it down as a prop, with the child short-circuiting the mount-time fetch when the prop is defined. The prop signature must be `undefined`-tolerant (`initialReported?: boolean`) so surfaces that *don't* pre-fetch (price cards, opportunity cards, marketplace cards, research cards) keep working without changes — strict back-compat protects every non-list surface from a regression while the list surface gets the batch optimisation. **Pair with two adjacent rules.** (a) **Audit your own diff against N+1 patterns the same week you ship it** — Sentry's flag landed within hours of the ReportButton on /community shipping; if we'd looked at the diff with N+1-vision before deploy, the batch-fetch would have shipped in the same commit. The cost of pre-shipping audit: 5 minutes. The cost of post-shipping fix: a separate commit + redeploy + Sentry triage cycle. (b) **The self-caused N+1 specifically generalises to ANY mount-time fetch in a row-renderer component** — including auth state, like state, follow state, save state. The /community page already batches likes + comments + parent posts + user-likes on the server; adding userReportedSet was extending the same pattern, not inventing a new one. The instinct should be "every per-row state lookup should be batched at the server query layer", not "we'll fetch per-row from the API and hope".

**72. Project-Scoped Skill Naming Is The Default — Skills Load On Description-Match Across All Installed Plugins**
First attempt at the Sentry-triage skill was named `sentry-triage`. Okoli flagged the conflict risk: "The name you gave to the file appears generic, won't it be conflicting?" Correct intuition. Skills load via fuzzy description-match across every plugin installed in a Cowork session — a generic name like `sentry-triage` will collide the moment a second project with Sentry installs a similarly-shaped skill, and the model has no principled way to disambiguate which project's config to use. The trigger language overlap is the failure mode: "run sentry triage" matches both, and the tie-break is non-deterministic. **Generalised rule.** Project-scoped skills carry the project's name as a prefix — `agroyield-sentry-triage`, `agroyield-daily-health`, `agroyield-migration-apply`, `agroyield-feature-flag-flip`. The prefix is load-bearing: it scopes the skill's knowledge baked-in (org slug, region URL, project slug, table names, column conventions) to one project unambiguously, and it disambiguates the trigger match for the model. **Pair with three adjacent rules.** (a) **Generic-named skills are fine for cross-project workflows** — `pdf`, `docx`, `pptx`, `xlsx`, `canvas-design` — but only when the skill's behaviour is genuinely project-agnostic. The litmus test: would installing this skill in a second project work without modification? If yes → generic is correct. If no → prefix it. (b) **The prefix should be the GitHub org or product slug**, not a person's name or a project codename — long-term identity is at the org level, not the contributor or the working title. AgroYield's prefix is `agroyield-`, not `okoli-` or `agri-platform-`. (c) **Even within one project, sub-domain prefixes earn their keep when skill counts grow** — at scale (10+ skills) we'd want `agroyield-sentry-triage` AND `agroyield-supabase-migrate` AND `agroyield-marketing-send`, not 10 skills under one undifferentiated prefix. The pattern compounds.

**73. Silent Insert Failure Is A Class-Of-Bug — Every Server-Write Boundary Needs Both `error` Destructuring AND A Row-Count Sanity Check**
The /api/report POST silent-insert-failure (post_type='price_report' rejected by `reports_post_type_check`, route returned 200 OK with a friendly user message, no row landed) is the same shape as the recurring-invoice silent-failure from Session 4 (HTTP 403 swallowed by a non-blocking `try/catch` after the primary write succeeded). Different surface, different root cause, identical user-visible failure mode: the system reports green when it's actually broken in a user-facing way. **Generalised rule.** At every server-write boundary, ALWAYS destructure `error` from the Supabase response AND check the row-count of the result. The error destructuring catches CHECK constraint violations, FK violations, RLS denials, and serialisation aborts. The row-count check catches the silent-no-op case where neither `data` nor `error` carry the failure — RLS policies that return 0 rows for a delete that "doesn't match", filters that match nothing on an UPDATE, race conditions where the row was deleted between the SELECT-where-exists check and the INSERT. The canonical write-handler shape becomes: `const { data, error } = await supabase.from(t).insert(row).select(); if (error) return 500; if (!data || data.length === 0) return 500 with row-count-zero-after-insert message; return 200 with data;`. **Pair with three adjacent rules.** (a) **Every write handler has a dual-fail-mode test in the test suite** — one that triggers the `error` path (e.g. duplicate key) and one that triggers the silent-no-op path (e.g. RLS policy that returns 0 rows). Both must produce 500s on the API and surface the actual cause. (b) **Add the row-count check as a `lib/db/safe-insert.ts` helper** so the canonical shape is one import and one function call instead of three boilerplate lines per route — eliminates the temptation to skip the check on routes that "feel safe". (c) **The CHECK constraint that caused this specific silent failure had been added out-of-band via Supabase Studio**, not via a tracked migration. The migration tracking table didn't know about it, so a fresh project clone wouldn't reproduce the constraint, which would mask the bug in dev. **Add a launch-gate audit query** that diffs `pg_constraint` against the migration-history-derived expected set — any constraint in pg_constraint that isn't in a tracked migration is either drift to formalise OR an out-of-band edit to revert. The schema is the source of truth only when it's wholly version-controlled.

### 23–27 April 2026 (Session 16b — Skill Naming, Daily Rituals, And The MCP Roadmap)

**74. The Daily-Health Ritual Is The Operational Equivalent Of The Pre-Rollout Runbook — Both Make Discipline Sustainable By Compressing Multi-Surface Manual Checks Into One Trigger**
Pre-launch we had a four-surface manual morning check: Slack #beta-alerts scan + Better Stack uptime + Vercel deploys + Supabase SQL counts. Each surface is 30-60 seconds. The whole check is 3-5 minutes when nothing's wrong, which sounds tractable. The failure mode is calendar-pressure: by week 2 of beta, the morning check skips Tuesday because something's urgent at 8am, then Wednesday because it skipped Tuesday, then it's a Friday-only ritual, then it's not happening. The 5-minute friction is the discipline killer. **Generalised rule.** Operational rituals that span ≥3 surfaces need a single-trigger consolidation skill (or runbook, or script) before they become regular practice. The skill is the load-bearing artefact — the discipline is downstream of the trigger ergonomics. The `agroyield-daily-health` skill collapses Sentry triage + 11-metric SQL + feature-flag check + moderation queue + auto-suspend watch list into one "morning check" invocation that returns one structured report; the user reads the report and decides what (if anything) needs follow-up. The skill is read-only by default — never auto-executes remediation — because operational rituals fail differently from operational actions: a check that auto-fixes is a check that hides root causes. **Pair with three adjacent rules.** (a) **Single-trigger skills earn their keep when the underlying surfaces are stable for ≥30 days** — premature consolidation locks in the wrong shape. The daily-health skill landed on day 1 of beta because the four surfaces (Sentry, DB counts, flag table, moderation reports) had been stable for ≥2 weeks of pre-beta operation. The Slack scan and Better Stack check are deferred to MCP-shaped tools because those connectors aren't installed yet — the skill flags them as manual-until-MCP and the workflow folds them in once the MCPs land. (b) **The skill's red-flag thresholds are explicitly placeholder until 14 days of beta traffic accumulate** — the launch-day numbers (5 community posts is yellow, 0 is red) are guesses calibrated against zero data; the skill notes this in Section F and flags revision after 14 days. Operational thresholds without empirical baselines are wrong by definition; ship them anyway with a self-scheduled revision date. (c) **The skill's own discipline depends on the user's discipline to invoke it daily** — there's no enforcement layer. The mitigation is to make the skill itself low-friction (single trigger, structured report, ≤5 min runtime) so the cost of daily use is genuinely lower than the cost of skipping. If the skill takes 15 minutes the discipline degrades; if it takes 3 minutes the discipline holds.

**75. The MCP Roadmap For An Operational Workflow Is A Living Document — Each MCP Lands, The Manual Check Folds Into The Skill, The Skill's Workflow Updates In-Tree**
The daily-health skill's Section "Manual checks (no MCP yet — do these in browser if no automation)" enumerates the surfaces that can't be queried via MCP today (Slack #beta-alerts, Better Stack uptime, Vercel deploys). Section "Roadmap when more MCPs land" pairs each manual check with the future MCP call that supersedes it (`mcp__slack__read_channel`, `mcp__betterstack__list_monitors`, `mcp__vercel__list_deployments`). When each MCP arrives, the skill's workflow updates from "open browser → check dashboard → eyeball" to "call MCP → parse response → fold into report" — and the manual section shrinks. **Generalised rule.** Operational skills that depend on connectors-that-don't-exist-yet should ship today with the manual fallback explicit AND the future MCP call explicitly named. The list of manual checks is a backlog of MCP work; making the dependency in-tree means the next session that lands a connector knows exactly which skills need updating. **Pair with two adjacent rules.** (a) **Don't gate the skill on the MCPs being available** — the skill is valuable today with the manual sections; ship it now and let it improve as the MCP ecosystem fills in. The cost of waiting for full MCP coverage is the cost of not having the skill at all for 30-60 days while connectors arrive. (b) **The MCP-roadmap section IS the BD pipeline for connector requests** — every manual check that doesn't have an MCP is a vote for that connector's existence. The aggregate of "manual check" sections across all skills in all projects is the prioritised connector backlog for the platform team.

06:30 WAT flag flip + 07:00 welcome email + 07:30 WhatsApp invite broadcast + 08:30 first-30-min monitoring window.
